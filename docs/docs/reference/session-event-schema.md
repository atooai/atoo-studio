# Session Event Schema

## How This Type Was Generated

The `SessionEvent` type in `src/events/types.ts` is a **programmatically derived superset**
of all message formats observed across agent session files. It was NOT hand-authored â€”
it was generated by scanning real session data.

### The Scan Script

To regenerate or extend the schema, run:

    python3 scripts/scan-session-schema.py [--dir ~/.claude/projects] [--output docs/schema-report.json]

The script:
1. Reads ALL `.jsonl` files in the Claude projects directory (recursive)
2. For each event, records:
   - The `type` field (discriminant)
   - Every field name and its observed value types (string, number, boolean, null, list, dict)
   - For `message.content` arrays: every content block `type` and its fields
   - For nested objects: their field shapes (e.g., `toolUseResult`, `data`, `usage`)
3. Outputs a JSON report with the complete field inventory per event type

### How To Add A New Agent Type (e.g., Codex, Gemini)

1. **Collect session data** â€” Run the new agent, generate some sessions
2. **Scan the data** â€” Point the scan script at the new agent's session files:
       python3 scripts/scan-session-schema.py --dir /path/to/codex/sessions
3. **Review the report** â€” Compare with existing `SessionEvent` variants
4. **Add new variants** to `src/events/types.ts`:
   - If the new agent uses event types not in the union, add them
   - If existing types have new fields, add them as optional
   - Prefix agent-specific fields with a comment: `// codex-specific`
5. **Add a writer** â€” Create `src/events/writers/codex-format.ts` that reconstructs
   the agent's native format from `SessionEvent[]`
6. **Add mapper cases** â€” Update `toWireMessages()` in `src/events/wire.ts` to handle
   new event types (map them to existing WireMessage types for display)

### Envelope Fields (common to conversational events)

These fields appear on `user`, `assistant`, `system`, and `progress` events:

| Field | Type | Description |
|-------|------|-------------|
| `uuid` | string | Unique event ID |
| `sessionId` | string | CLI session UUID |
| `parentUuid` | string\|null | Previous event in conversation chain |
| `timestamp` | string | ISO-8601 timestamp |
| `type` | string | Event type discriminant |
| `cwd` | string | Working directory |
| `gitBranch` | string | Current git branch |
| `version` | string | Claude CLI version |
| `slug` | string | Session slug |
| `userType` | string | Always "external" currently |
| `isSidechain` | boolean | Whether this is a subagent event |
| `agentId` | string? | Subagent identifier |

### Content Block Types (inside message.content arrays)

**Assistant content blocks:**
- `text` â€” `{ type, text }`
- `thinking` â€” `{ type, thinking, signature }`
- `tool_use` â€” `{ type, id, name, input, caller? }`

**User content blocks:**
- `text` â€” `{ type, text }`
- `tool_result` â€” `{ type, tool_use_id, content, is_error? }`
- `image` â€” `{ type, source }`

### toolUseResult Variants (on user events)

The `toolUseResult` field on user events carries tool output in various shapes:

| toolUseResult.type | Fields | Tool |
|---|---|---|
| `text` | file `{content, filePath, numLines, startLine, totalLines}` | Read |
| `create` | content, filePath, originalFile, structuredPatch | Write |
| `update` | content, filePath, originalFile, structuredPatch | Edit |
| `unknown` | content, prompt, status, totalDurationMs, totalTokens, totalToolUseCount, usage | Agent |
